That time I got stuck on a JavaScript problem

A Simple Problem

Well I don't know about you, but I had some big blocks on the fizzbuzzed challenge! And as predicted by what Joseph was saying in the videos, I totally got blocked on how functions work (particularly the words inside the brackets!). Even after watching the video explaining it and thinking I understood, it's only now I realise that I still didn't quite get it!

For example, I didn't realise that in the function super_fizzbuzz (arr) {some instructions for the function} that it doesn't matter what (arr) is named and that it doesn't mean anything until you call (or in other words actually use) the function. It is not a string, it is not an array. It's just nothing until it is called. This was hard for me to get my head around because it felt like I was having to think in a non-linear order. The hardest part of learning JavaScript has been getting used to the non-linear order of things and how it feels like things are jumping back and forth all over the place. And maybe once all this information is grounded I'll be saying it's so linear and logical. Who knows?!

So how did I solve this conundrum? Well because I already had the code (by way of copying and pasting from the internet and referencing the sample solutions to double check), I knew I just needed some help by peers/coaches to explain how it was working to me. I reached out to my facilitator who explained it in one way, and then when I got confused by this same thing in the next kata I reached out to my peers who explained it in a different way. It took about 3-4 people explaining this concept for it to finally sink in enough for me to be able to explain it to myself (sort of like the rubber ducky method but talking to myself with my flatmates around me)!

Throughout this process I felt quite lost and when you really don't understand something, it's hard to even ask the right questions. I find this the most frustrating part of the learning process, when you know so little, that even asking a question feels like you need some knowledge first to be able to ask a good one! But I think in the end, through a combination of problem solving techniques it is possible to get there!

An Elegant Solution

Elegant isn't the first word I would use to describe my problem solving. But I would say I'm proud of the learning process I took in the "Built-in Methods" kata. By this point, I had gained some experience and now felt like I could approach a problem and ask better questions. An example of this was writing the code for the capitalise function. In this instance I knew this would likely be a commonly asked question so I simply googled it, found an answer on Stackoverflow and copied and pasted their code into mine and adjusted any word differences as necessary. From memory, this didn't work straight away so I think I checked the sample solution and then researched the .substring function on MDN. I found MDN really useful for diving deep into the particulars of the code to deepen my understanding.

Once I had fixed my code so it passed the test, I wanted to see for myself that it was working so I console.logged it. It did capitalise the beginning of each word! Yay! So then I messed around with the capitalise function to see what would break it or work differently so that I could get a good understanding of the function. For example I wanted to understand the .split function so I played around with the parameter it was being split by being " " or "," or ";" and console.logging each time to see the result. I have to say it was satisfying to understand what was happening at each part and stage of the capitalise function!

Throughout this process, I felt calm and trusted I could learn and understand the code required to solve the problem. I learnt that there is always a solution out there, someone has probably explained it well and there is no one way of writing code that works. But if it works then hooray!
